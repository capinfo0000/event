# 運用手順

事前決済（前払い）でイベント参加費を集める際の、実際の運用フローです。

## 1. イベントを登録する

`config/events.json` に新しいイベントを追加します（`id` は重複しない英数字）。

## 2. 申込URLを参加者に告知する

- 開発時: `http://localhost:8000`
- 公開時: デプロイ先のURL（例: `https://あなたのドメイン/`）

LINE・メール・SNS などで申込ページのURLを共有します。参加者は一覧から
「申し込む（前払い）」を押し、Stripe の安全な画面でカード決済します。

## 3. 申込状況を確認する（＝参加者名簿）

[Stripe ダッシュボード](https://dashboard.stripe.com/payments) の「支払い」一覧が
そのまま参加者名簿です。各支払いに以下が記録されています。

- 参加者の氏名（custom field「参加者のお名前」）・メール・電話
- どのイベントか（メタデータ `event_id` / `event_name`）
- 金額・支払い状況

「支払い」一覧で `event_name` などで検索・絞り込みができます。

## 4. ドタキャン・欠席への対応（本アプリの核心）

**前払い済みなので「取り立て」は不要です。** キャンセルの連絡が来たら、
[`docs/キャンセルポリシー`](../public/policy.php) の返金率に従って Stripe 上で操作します。

- 全額/一部返金する場合: ダッシュボードで対象の支払いを開き「返金」ボタン → 金額を指定。
- 返金しない場合（規定により）: 何もしません。すでに受領済みです。

> これにより「規約にキャンセル料を書いても回収できない」問題が構造的に解消します。
> 先にお金を預かっているため、返金しない＝実質的なキャンセル料の回収になります。

## 5. 当日受付

Stripe ダッシュボードの支払い一覧（＝名簿）を見ながら受付します。
氏名・メールで本人確認できます。

## 返金ポリシーの変更

返金率の文面は `public/policy.php` を直接編集します。決済画面のボタン直上にも
要約メッセージが出ます（`public/checkout.php` の `custom_text.submit.message`）。

## テストから本番への切り替え

1. テストモードの `sk_test_...` で一通り動作確認（テストカード `4242 4242 4242 4242`）。
2. 問題なければ `.env` を本番キー `sk_live_...` に変更。
3. Stripe 側で入金先口座・本人確認（KYC）を完了させる。
4. （任意）本番URLの `webhook.php` を Stripe ダッシュボードの Webhook に登録し、
   署名シークレット `whsec_...` を `.env` に設定。

## 注意

- `.env`（APIキー）と `logs/*.log`（個人情報）は **Git にコミットしない**設定です（`.gitignore`）。
- 特定商取引法に基づく表記が必要な場合は、別途ページを用意してください（規模・形態により要否が異なります）。
