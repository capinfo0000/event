# 開発の記録（なぜ作り・何に悩み・どう改善し・どうなったか）

本アプリ開発の経緯を後から追えるようにまとめた作業記録です。
「背景／目的」「設計方針」「開発の流れ」「直面した課題と対応」「最終的な仕様」「今後の課題」で構成しています。

> 補足：本プロジェクトは途中で一度「マルチテナント型SaaS」まで広げたのち、
> 運用のしやすさを優先して **単独運営者向けの自己ホスト型** に再構成しました。
> その経緯も含めて記録しています。

---

## 1. 背景・目的

小規模イベント（BBQ・お花見・スポーツ等）では、当日欠席・ドタキャンによって
**「キャンセル料を後から取り立てられない」** という問題が起きやすい。
これを **参加費を先に支払ってもらう（事前決済）** ことで解消するのが出発点。

加えて「当日は現金で払いたい人もいる」ため、**当日支払い**も選べるようにした。

---

## 2. 設計方針（一貫して守ったこと）

- **カード情報を持たない**：決済は Stripe Checkout（Stripeホストの決済ページ）で行い、
  カード番号等はサーバーを通さない。
- **参加者名簿はDBに持たない**：名簿は Stripe（事前=Checkoutセッション、当日=顧客）から都度取得。
  個人情報の保管箇所を最小化する。
- **DBに持つのは最小限**：運営アカウント・イベント定義など運営データのみ（SQLite）。

---

## 3. 開発の流れ（段階）

| 段階 | 内容 |
|---|---|
| 初期 | 単一運営者向けの事前決済アプリ（名簿はStripeから取得、返金・CSV） |
| 当日支払い対応 | 「当日支払い」を選択可能に。当日分は課金なしのStripe顧客として記録 |
| 名簿の強化 | 当日支払いに「受領」確認、事前決済者に「事前決済済み」表示 |
| マルチテナント化 | SQLite導入、アカウント/ログイン、Stripe Connectで主催者ごとに入金先を分離 |
| 収益化（プラン） | 開催月あたりのイベント数で段階化した料金プラン＋Stripeサブスクで運営者へ課金 |
| 運用機能 | パスワード再設定・アカウント設定・ログイン試行制限・Cookie堅牢化・法務ページ・確認メール |
| オープン登録 | 招待制をやめ、メール＋パスワードで誰でも登録できるように |
| **単独運営者向けに再構成** | Connect・プラン課金を取り外し、運営者自身のStripeキーで決済する自己ホスト型へ。UIをサイドバー＋グラフに刷新 |
| 配布 | ローカル実行手順の整備、ソースZIP・共有サーバー設置用ZIP（vendor同梱）の作成 |

### なぜ単独運営者向けに戻したか
- マルチテナント＋Connect＋サブスク課金は機能としては成立したが、
  **運用（Connect申請・Webhook・プラン課金・法務）が重く**なりがち。
- 「自分（自団体）のイベントを回せれば十分」という用途に対しては、
  **運営者が自分のStripeキーを1つ入れるだけ**で動く自己ホスト型のほうがシンプルで安全。
- そこで Connect とプラン課金の導線を外し、決済は運営者自身のアカウントで直接行う形にした。

## 4. 直面した課題と、その対応

### 4.1 「DBを持たない」方針と、保存が必要な情報の両立
- 課題：参加者個人情報はできるだけ持ちたくないが、ログインやイベント定義には永続化が要る。
- 対応：**参加者名簿はStripe由来のまま**にし、DB（SQLite）には運営アカウントとイベント定義だけを保存。

### 4.2 当日支払い・出席などの状態をDBなしで持つ
- 課題：「受領（集金）済み」「出席済み」をどこに記録するか。
- 対応：**Stripeの顧客（Customer）の metadata** に保存（`collected` / `attended`）。
  事前・当日いずれも顧客が存在するため、出席は両方式で共通の仕組みにできた。

### 4.3 入金先の扱い（Connect → 自己ホストへ）
- 経緯：当初はマルチテナント化のため Stripe Connect で「主催者ごとに入金先を分離」した。
- 課題：単独運営なら、わざわざ接続アカウントを介すより**運営者自身の口座へ直接**が簡単で確実。
- 対応：Connectを外し、決済APIは接続アカウント指定なし（自アカウント）で作成する形へ統一。
  （`stripe_opts()` は残し、将来再導入できる余地は確保。）

### 4.4 プラン上限の「数え方」（→ 最終的に撤去）
- 経緯：収益化のため「開催月あたりのイベント数」で段階課金を実装
  （日付から開催月 `YYYY-MM` を抽出して同月件数で判定。上限到達時は課金画面へ誘導）。
- 変更：単独運営者向け再構成に伴い、プラン課金・上限の導線は取り外した
  （開催月を判定する `event_month()` 等の仕組みは、入力チェック等で引き続き活用）。

### 4.5 ローカルではStripe通信ができない
- 課題：開発環境ではStripeへ実通信できず、決済系を実地テストできない。
- 対応：**Stripe非依存の部分（認証・DB・イベント管理・画面描画・パスワード再設定）をローカルで検証**。
  Stripe実フローは利用者側のテスト鍵での確認事項として明記。

### 4.6 ローンチ前に不足していた“周辺機能”
- 課題：パスワード再設定が無い／法務表記が無い／当日申込に通知が無い等の穴。
- 対応：**パスワード再設定・アカウント設定・ログイン試行制限・セッションCookie堅牢化・
  特商法/規約/プライバシーのテンプレート・当日申込の確認メール** を追加。

### 4.7 ローカル環境構築（Windows）
- 課題：利用者環境のPHPが **7.4** で要件（**8.1+**）を満たさず `composer install` も失敗。
- 対応：XAMPPを更新して **PHP 8.2** を用意。拡張有効化とコマンド手順を案内し、起動まで到達。

### 4.8 配布・デプロイ
- 課題：共有レンタルサーバーで composer を使わず動かしたい。
- 対応：**`vendor/` 同梱の設置用ZIP** と日本語の設置手順を用意。
  セキュリティ上、**公開フォルダを `public/` にする**（`src`/`.env`/`data`を外に置く）よう明記。

## 5. 最終的な仕様（現在の構成：単独運営者向け）

- 決済：**運営者自身の Stripe アカウント**（`.env` の `STRIPE_SECRET_KEY`）で直接。Connect不使用。
- 運営者：オープン登録／ログイン／パスワード再設定／アカウント設定
- ダッシュボード：申込推移・支払い方法内訳のグラフ、Stripe設定状況、公開リンク（サイドバーUI）
- イベント：登録/編集/削除、事前/当日の金額・受付可否を個別設定、申込リンク・公開一覧（`o.php`）
- 申込：人数・氏名等、事前決済（クレカ）or 当日支払い（現金）
- 名簿：返金（全額/一部）、当日「受領」、全員「出席」、CSV出力（Stripeから取得）
- 当日申込には受付確認メールを自動送信
- 法務：特商法・規約・プライバシー（テンプレート）
- データ：SQLite（運営アカウント・イベント定義のみ）。名簿・決済はStripeが正

---

## 6. 技術構成

- 言語/実行：PHP 8.1+（開発は組み込みサーバー `php -S`、共有サーバーにも設置可）
- 決済：Stripe（Checkout・当日分は課金なしCustomer）
- データ：SQLite（`pdo_sqlite`）
- 認証：セッションログイン（CSRF対策・Cookie堅牢化・ログイン試行制限）
- メール：`mail()` ＋ `logs/mail.log`（本番はSMTP等へ差し替え推奨）
- UI：`public/assets/app.css` ＋ Chart.js（ダッシュボードのグラフ）
